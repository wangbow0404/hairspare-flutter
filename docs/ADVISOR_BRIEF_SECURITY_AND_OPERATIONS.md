# HairSpare 서비스 개요 및 보안·운영 자문 요청

> **대상**: 서버/보안 실무자 (컴공 전공, 네트워크·비용·보안 관점 자문)  
> **목적**: 런칭 전 준비사항, DDoS 대응 방향, API 보안, 실제 배포·운영 리스크, 그리고 현재 구조에서 꼭 잡아야 할 부분에 대한 조언을 구하기 위함.

---

## 1. 서비스가 만들고자 하는 것

### 1.1 서비스 정의

**HairSpare**는 **미용실 스페어(헤어 디자이너) 급구 해결 플랫폼**입니다.

- **미용실(Shop)**: 급하게 필요한 인력을 공고로 모집하고, 지원자 승인·스케줄·정산을 관리
- **스페어(Spare)**: 공고에 지원하고, 예약금(에너지) 결제·출근 체크·스케줄·채팅·포인트를 사용
- **관리자(Admin)**: 회원·공고·결제·에너지·노쇼 등 플랫폼 전반 관리

### 1.2 런칭 목표 (자문 맥락)

- **첫 달 MAU 1,000명 이상**
- **재방문율 30% 이상**

이 수준을 목표로 하고 있으며, 그 **이전 단계에서** 보안·인프라·운영 측면에서 미리 준비해야 할 것이 무엇인지 조언을 받고 싶습니다.

### 1.3 계획된 기능 범위 (스토어까지)

| 구분 | 내용 | 현재 상태 |
|------|------|------------|
| **스페어** | 공고 목록/지원, 에너지(예약금) 잠금·반환, 스케줄, 출근 체크인, 채팅, 포인트, 찜, 프로필, 공간대여, 교육·챌린지·커넥트 진입 | 화면·플로우 대부분 구현 |
| **미용실** | 공고 등록·관리, 지원자 승인/거절, 스케줄, 채팅, 포인트, 인력 검색, 프로필, 공간대여 관리, VIP 현황 | 화면·플로우 대부분 구현 |
| **관리자** | 대시보드 통계, 회원/공고/결제/에너지/노쇼 관리 | 일부 구현, 일부 Next.js 의존 |
| **스토어** | 상품·카테고리·장바구니·주문 (API 경로는 Gateway에 존재) | 백엔드 라우팅만 있고, Flutter 스토어 화면·연동 미구현 |
| **결제·알림** | 결제 플로우, 푸시/알림 | Phase 6에서 스텁 수준 계획 |

**Phase 요약** (통합 개발 플랜 기준):

- Phase 0~1: 스케줄 체크인/확정, 에너지 잠금·반환, 지원 승인/거절
- Phase 2~4: 찜 백엔드, 리뷰/ThumbsUp, 채팅 보강, Admin 실데이터 연동
- Phase 5~6: Admin API 정리, 결제/알림/스토어 스텁 이상 구현

## 2. 지금까지 만든 것 (구현 현황)

### 2.1 클라이언트 (Flutter)

- **역할**: 스페어 / 미용실 / 관리자(일부)
- **플랫폼**: iOS·Android·Web (Flutter 하나로 대응)
- **구현된 화면 예시**  
  - 스페어: 홈, 공고 목록·상세, 지원, 스케줄, 출근 체크, 에너지·포인트, 채팅·메시지, 찜, 프로필, 공간대여, 검색·알림 목록 등  
  - 미용실: 홈, 인력별(스페어 목록), 공고 목록·상세·등록, 지원자 관리, 스케줄, 포인트, 채팅, 프로필, VIP 현황 등  
  - 스토어·교육·챌린지·커넥트: 카테고리 진입만 있고, 실제 스토어 페이지 등은 미구현
- **참고 문서**: `docs/PROJECT_STATUS.md`, `docs/SHOP_PAGES_STRUCTURE.md`

### 2.2 백엔드 (현재 구조)

- **API Gateway (FastAPI, 포트 8000)**  
  - 모든 클라이언트 요청이 여기로 들어옴  
  - 인증(로그인/회원가입 등) 일부 경로만 공개, 나머지는 (의도된) 인증 미들웨어 비활성 상태
- **마이크로서비스 (FastAPI, 각각 단일 포트)**  
  - Auth(8101), Job(8103), Schedule(8104), Chat(8105), Energy(8106)  
  - Store/Cart/Order/Payment/Notification 등은 Gateway에 라우팅만 있고, 해당 서비스는 미구현·미연동 가능
- **DB**: PostgreSQL
- **관리자 API**:  
  - `/api/admin/stats`, `/api/admin/users` 등은 Gateway에서 DB 직접 조회  
  - `/api/admin/jobs`, `/api/admin/payments`, `/api/admin/energy`, `/api/admin/noshow` 등은 **Next.js(3000) 프록시** 의존

### 2.3 API 경로 요약 (전체 – 보안·비용 자문용)

Gateway를 통과하는 주요 경로입니다. **전체 API**에 대한 보안·설계 자문을 받고자 합니다.

| 영역 | 경로 예시 | 비고 |

```
            │
            ▼
[ API Gateway :8000 ]  ← 단일 진입점, CORS·라우팅·(향후) 인증
            │
   ┌───────┼───────┬──────────┬──────────┬────────────┐
   ▼       ▼       ▼          ▼          ▼            ▼
[ Auth ] [ Job ] [ Schedule ] [ Chat ] [ Energy ] [ Next.js ]
 8101    8103      8104       8105      8106        3000(admin 일부)
   │       │         │          │          │
   └───────┴─────────┴──────────┴──────────┘
                        │
                        ▼
                 [ PostgreSQL ]
```

- **문제점·갭** (ARCHITECTURE.md 등 기준):  
  - 관리자 일부는 Next.js 의존 → 배포·보안 일원화 어려움

  - Store/Cart/Order 등 서비스는 라우팅만 있고 미구현 가능

## 4. 현재 구조에서의 문제·개선 필요 사항

(보안·운영 자문과 맞닿는 부분만 요약)

| 구분 | 내용 |
|------|------|
| **인증·권한** | 관리자 API 보호 미적용, 스페어/미용실 토큰 검증 정책 불명확 |
| **단일 진입점** | Gateway 하나에 트래픽 집중 → DDoS·비용 스파이크 시 영향 집중 |
| **Admin 이원화** | Next.js 프록시와 DB 직접 혼재 → 운영·감사·보안 정책 통일 어려움 |
| **스토어·결제** | 스토어 페이지·결제 플로우 미완성, 결제·개인정보 구간 보안 검토 필요 |
| **배포·운영** | 실제 프로덕션 배포(HTTPS, 도메인, 방화벽, 로그·모니터링) 미정리 |

## 5. 자문을 구하고 싶은 부분 (질문 정리)

아래 **5가지**에 대해, “어느 수준까지 준비해야 하는지”와 “실무자 도입이 필요한지 vs 자체 대응이 가능한지”에 대한 조언을 받고 싶습니다.

---

### 5.1 런칭 전 보안·인프라 준비 (MAU 1,000 / 재방문 30% 목표)

- **목표**: 첫 달 MAU 1,000, 재방문율 30% 이상을 목표로 하고 있음.
- **질문**:  
  - 이 수준을 **넘기기 전에** 미리 갖춰 두어야 할 보안·인프라·운영 조치가 있는지?  
  - (예: 인증 강화, Rate Limit, 로그·모니터링, 백업·복구, 비용 알림 등)  
  - “이것만은 반드시 해두고 오픈하라”는 항목이 있다면 알려주시면 좋겠음.

---

### 5.2 DDoS·과도한 트래픽 대응 방향

- **상황**: 네트워크 트래픽·비용은 예측이 어렵고, DDoS 시 비용이 기하급수적으로 늘어날 수 있다는 점을 인지하고 있음.
- **질문**:  
  - DDoS(및 일반적인 트래픽 스파이크) 대비는  
    - **보안 실무자(외부 자문/도입)**를 쓰는 것이 맞는지,  
    - 아니면 **우리 팀이 할 수 있는 범위**(예: CDN·WAF·Rate Limit 설정 등)가 있는지  
    **역할 분담 관점**에서 어떻게 나누는 것이 좋은지 조언을 구하고 싶음.  
  - “이 규모(MAU 1,000 수준)에서는 여기까지는 직접, 그 이상은 전문가” 같은 기준이 있다면 그 경계도 알고 싶음.

---

### 5.3 전체 API 보안

- **범위**: 위 **섹션 2.3**에 정리한 **전체 API** (인증, 공고, 스케줄, 채팅, 에너지, 찜, 스토어·결제·알림, 스페어, 공간대여, 관리자)에 대한 보안 관점 조언.
- **질문**:  
  - 인증·인가, Rate Limit, 입력 검증, 로깅·감사, 민감 데이터(결제·개인정보) 처리 등  
    **우선순위를 두고 손봐야 할 항목**이 무엇인지,  
  - 그리고 **지금 구조(Gateway + MSA)**에서 특히 취약하거나 개선이 시급한 부분이 있다면 알려주시면 함.

---

### 5.4 실제 배포·운영 시 예상 리스크

- **질문**:  
  - 현재처럼 **단일 Gateway + 여러 서비스 + DB** 구조를 **실제 서비스로 배포·운영**할 때,  
    **보안·안정성·비용** 측면에서 **문제가 생기기 쉬운 부분**이 무엇인지 알려주시면 함.  
  - (예: SSL/TLS, 방화벽, 서비스 간 통신, DB 노출, 로그·모니터링 부재, 스케일링 한계 등)  
  - “이걸 해결하지 않고 오픈하면 위험하다”는 항목을 구체적으로 짚어주시면 좋음.

---

### 5.5 전체 구조·우선순위 종합 조언

- **질문**:  
  - **개발 중인 앱과 현재 구조**를 위 내용처럼 이해했다는 전제 하에,  
    - **구조적으로 문제인 부분**,  
    - **지금부터라도 꼭 잡고 가야 할 것**(보안뿐 아니라 운영·비용 포함),  
    - **우선순위**(예: 오픈 전 필수 vs 오픈 후 단계적)  
  을 정리해 주시면, 그에 맞춰 준비하고 싶음.
