# 결제 아키텍처 설계 (모델·스페어·샵 매칭 정산 + 플랫폼 수익)

> 상태: **설계/계획 단계 (코드 미착수)** · PG는 추후 연동 예정
> 작성 배경: 모델↔샵/스페어 매칭 후 대화가 끝난 시점의 결제 로직을 어떻게 짤지 구상.
> ⚠️ 본 문서는 구현 설계 관점이며 법률 자문이 아님. 전자금융업 등록 기준 등은 변호사·세무사 확인 필요.

---

## 0. 지금 코드의 실제 상태 (조사 결과)

| 구분 | 상태 |
|---|---|
| `payment-service` | 껍데기(health check 18줄만). 로직 0 |
| `/api/payments` (GET/POST) | 스텁. POST는 "결제 서비스 준비 중입니다" 202만 반환 |
| 에너지(Energy) 시스템 | ✅ 동작. `EnergyWallet`(잔액) + `EnergyTransaction`(charge/spend/refund, state: available/locked/returned/forfeited). 1에너지=1000원 |
| 에너지 "충전"(`/api/energy/purchase`) | ⚠️ 잔액만 숫자로 더해줌. **실제 PG 연동 없음** — 진짜 돈은 안 들어옴 |
| 보증금 개념 | ✅ 설정에 `modelDepositAmount`(3만원) 존재. state의 locked/returned/forfeited = 홀드→환불→몰수 패턴 |
| 노쇼 신고(`NoShowReport`), 근무 체크인/완료(`WorkSchedule.check_in/out`) | ✅ 이미 있음 |
| 채팅(`Chat`/`Message`) | ✅ 이미 있음. 메시지 "종류" 확장 여지 있음 |

**요약:** 결제의 *뼈대*(원장·보증금 상태·노쇼 처리)는 있으나 **실제 현금이 들어오는 PG 연동이 통째로 비어 있음.** 핵심 작업 = PG 선택 + 이걸 채팅 흐름에 얹기.

## 1. 우리 회사의 "두 개의 역할" (이중 포지션)

한 회사가 거래에 따라 다른 모자를 씀. 정상이고 합법(네이버·쿠팡·배민 동일).

| 상황 | 우리 역할 | 리스크 |
|---|---|---|
| 급구·하이패스·프리미엄·구독·채팅 애드온 | **판매자 (통신판매업자)** — 우리 서비스를 우리가 직접 팜 | 낮음. 제3자 분쟁 거의 없음 |
| 모델↔샵/스페어, 샵↔스페어 정산 | **중개자 (통신판매중개자)** — 남들 거래를 연결 | 중개자 고지로 관리 |

- "중개자 포지션 고정"이 아니라 **거래마다 역할이 다름.** 거래 화면마다 현재 역할을 고지.
- **수수료를 받아도 중개자 지위는 안 깨짐** — 중개자는 원래 수수료로 수익을 냄.

---

## 2. ⭐ 리스크 최소화의 핵심: "돈이 어디로 흐르냐"

수수료 유무가 아니라 **자금 흐름 구조**가 리스크를 가름.

- ❌ **위험**: 결제금이 우리 계좌로 다 들어왔다가 수수료 떼고 수취인에게 재송금 → 남의 돈 보유 → 전자금융거래법상 등록·인가 이슈(대금예치업 등)
- ✅ **안전**: PG **분할정산(하위몰/파트너 정산)** — 결제 순간 PG가 수취인에게 (금액-수수료), 우리에게 수수료를 **각각** 송금 → 우리 계좌에 남의 돈이 고이지 않음

**원칙: "수수료는 받되, 정산금은 우리 계좌를 거치지 않게."** PortOne·Toss 등 요즘 PG는 분할정산 기본 지원.

### 리스크 최소화 4원칙
1. **중개자 포지션 고지** — 사람 간 거래엔 "본 앱은 중개자이며 거래 당사자가 아님, 책임은 당사자에게" 명시
2. **앱이 자금 미보유** — PG 분할정산으로 앱 계좌 미경유
3. **상호동의·정책 기록** — 금액/방향/취소·환불·노쇼 정책을 결제 시점에 양쪽 동의(체크) 후 스냅샷 저장 (분쟁 증거)
4. **금액에 앱 비관여** — 금액은 100% 사용자 합의값. 앱은 수수료 규칙만

> 그래서 사람 간 정산은 "에너지 충전"(앱이 선불금 보유 → 리스크↑)이 아니라 **PG 직결**이 유리.
> 에너지는 공고 노출·급구·하이패스 등 **앱 내부 소비용**으로 유지.

---

## 3. UX 결정: 채팅엔 "요청 카드"만, 결제는 전용 화면

- ❌ 채팅창에 결제창 통째로 임베드 — 채팅은 실시간(WebSocket)이라 PG 화면과 생명주기 충돌, 보안·재시도 지저분
- ✅ **채팅엔 "결제요청 카드"(트리거)만, 실제 PG는 격리된 전용 결제화면** (당근페이·번개장터 안전결제 패턴)
  - 트리거가 대화 맥락에 남음(누가 언제 요청했는지 기록)
  - 결제 상태(요청→완료→환불)를 카드에 실시간 반영

---

## 4. 단가 공개 정책 (프라이버시 배려)

모델 단가를 프로필에 강제 공개하면: ①프라이버시 부담 ②앱이 가격을 "전시"하면 가격 결정 주체처럼 보여 책임 리스크↑ ③경직적.

**➡️ 하이브리드 추천:**
- 프로필 기본값 = **"협의"** (금액 비표시)
- 원하는 사람만 **희망 금액/범위 선택적 표시** (비공개 유지 가능)
- **최종 금액은 항상 채팅 "결제요청 카드"에서 양쪽 명시 합의**

---

## 5. 결제요청 카드 (핵심 컴포넌트)

채팅 시스템 메시지 타입을 확장해 카드로 렌더. 카드에 담기는 정보:

| 항목 | 값 |
|---|---|
| **방향** | payer → payee (양방향) |
| **목적** | 모델페이 / 시술·재료비 / 보증금 / 일당 정산 |
| **금액** | 합의값 |
| **정책** | 취소·환불·노쇼 정책 스냅샷 (양쪽 동의 체크) |

흐름: 한쪽이 카드 생성(요청) → 상대가 수락 → 전용 결제화면(PG) → 결제 → 상태 실시간 갱신.

---

## 6. 거래 유형별 흐름

### 6-1. 모델 ↔ 스페어(디자이너)/샵

```
① 매칭 성사(하트 수락 / 날짜 확정) → 채팅방 열림 (구현됨)
② 채팅에서 조건·금액 협의
③ 한쪽이 [결제 요청] → 결제요청 카드 (방향+목적+금액)
④ 상대 수락 → 전용 결제화면(PG) → 결제
⑤ PG 분할정산 (수취인 / 우리 수수료 각각, 앱 미경유)
⑥ 시술 완료(체크인) → 확정  /  노쇼(신고) → 정책대로
```

방향에 따라 payer가 바뀜(양방향):
- 유명 모델 → 디자이너가 **모델페이** 결제 (디자이너 = payer)
- 신입 모델 → 모델이 **시술·재료비** 결제 (모델 = payer)

**결제 시점: 즉시 결제 + 분할정산 (단순).**
- ~~예치(에스크로)/보증금 홀드~~ → **폐기.** 노쇼 방지는 돈이 아니라 **정책+블랙리스트**로 처리(아래 11절). 결제는 "그냥 받기"로 단순화 → 앱이 돈을 홀드하지 않아 리스크·복잡도 모두↓.

### 6-2. 샵 ↔ 스페어 (공고-지원)

- 이미 공고·지원·근무일정(`WorkSchedule`, `JobApplication.energy_locked`) 뼈대 존재
- 기본 방향: 샵이 스페어에게 **일당** 지급
- 같은 "결제요청 카드" 구조 재활용, 목적 = "일당 정산"
- 완료 트리거 = `WorkSchedule` 체크인/완료, 노쇼 = `NoShowReport`

## 7. 상태머신 (공통) — 단순화

```
requested(요청) → agreed(합의) → paid(결제완료) → completed(완료) / cancelled(취소·환불)
```
- 완료: `WorkSchedule` 체크인/완료
- 노쇼는 이 결제 상태머신이 아니라 **정책/블랙리스트(11절)**로 별도 처리 (돈 몰수 없음)
- 홀드/예치(locked/forfeited) 단계 없음 → 즉시 결제라 단순

## 8. PG 비종속 설계 (추후 연동 대비)

지금 특정 PG를 정하지 않아도 됨. **`PaymentProvider` 공통 인터페이스**로 도메인 설계 →
나중에 어떤 PG든 어댑터만 교체.

인터페이스가 가져야 할 것:
- 결제요청 생성 (createPayment)
- 승인/확정 (capture)
- 부분/전체 취소·환불 (cancel/refund)
- 웹훅 서명 검증 (verifyWebhook)
- 분할정산 정보 (payee/commission split)

지금 비어있는 `payment-service` / `/api/payments` 스텁이 이 자리.

## 9. 재활용 가능한 기존 자산 (신규 구현 최소화)

- 채팅(`Chat`/`Message`) → 메시지 종류만 확장 → 결제 카드
- 근무 체크인/완료(`WorkSchedule`), 노쇼 신고(`NoShowReport`)
- 가격 설정(`modelDepositAmount` 등 비즈니스 설정)
- `payment-service` / `/api/payments` 스텁 → 채우기
- 매칭(하트 수락 / 날짜 확정) → 채팅방 생성까지 이미 연결됨

## 11. 노쇼 방지 = 보증금(돈) 대신 "정책 + 블랙리스트"

노쇼 방지를 보증금으로 하지 않기로 결정. 대신 **누적 위반 시 계정 탈퇴 + 재가입 차단(블랙리스트)**.

### 11-1. ⚠️ 재가입 차단은 "주민등록번호"가 아니라 "CI/DI"로

- **주민등록번호 저장 금지**: 개인정보보호법 제24조의2 — 법령 근거 없이는 수집·저장 위법. 블랙리스트 대조용으로 주민번호를 쓰면 오히려 앱 리스크 폭증. **절대 사용 금지.**
- 이름+전화 대조도 약함 (개명·번호변경·명의도용으로 우회, 동명이인 오차단).
- **합법·강력한 방법 = CI/DI (본인인증 부산물)**:
  - **CI**(연계정보): 사람마다 고유한 일방향 암호화 값 (주민번호 아님)
  - **DI**(중복가입확인정보): 서비스 내 사람 구분값 — "재가입 차단" 목적으로 만들어진 값
- 우리 앱은 이미 PASS 본인인증(`Verification` 모델: `identity_name/phone/birth_date`, `pass_transaction_id`)을 쓰지만 **CI/DI(사람 고정값)는 저장 안 함** → `Verification`에 `ci_hash`(또는 `di_hash`) 필드 추가 필요.

### 11-2. 블랙리스트는 별도 테이블 (탈퇴 후에도 생존)

`User`는 탈퇴 시 (백그라운드) 삭제될 수 있으므로 블랙리스트는 독립 테이블이어야 재가입을 막을 수 있음.

```
[신규] BlacklistEntry
  - ci_hash          # 본인인증 CI 해시 — 재가입 시 대조 키
  - reason           # noshow_accumulated | chat_policy_violation | other
  - banned_at, banned_by(admin)
  - released (bool)  # 관리자가 풀어주면 true → 재가입 허용
```
가입(본인인증) 시 새 CI가 블랙리스트에 있고 `released=false`면 가입 차단.

### 11-3. ⭐ 단계적 적용 (지금은 차단 스위치 OFF)

**현재 PASS 본인인증 API 미수령 → CI를 못 받음.** 지금 실제 차단을 켜면 신규 가입 불가/테스트 불가.
→ **구조(테이블·필드·관리자 화면)는 지금 만들되, 실제 차단은 플래그로 꺼둠.**
- 비즈니스 설정에 `identityVerificationEnforced`(기본 **false**) 추가
- false인 동안: 본인인증·CI 대조 스킵, 계정 자유 생성 (테스트 가능)
- 본인인증 API 붙는 날 → true로 전환 → 차단 작동
- 관리자 블랙리스트 화면·수동 등록/해제는 플래그와 무관하게 먼저 테스트 가능

### 11-4. 누적 위반 → 자동 탈퇴 임계값

- 소스: 노쇼 횟수(`NoShowReport`) + 채팅 경고(`User.contact_warning_count` — 이미 존재)
- 임계값을 **비즈니스 설정**에 추가 (예: `noShowBanThreshold`=3) → 관리자 조정
- 도달 시: 자동 정지/탈퇴 + `BlacklistEntry` 생성 (사유 기록)

### 11-5. 관리자 페이지 배치

- **추천: 기존 "회원관리"에 "블랙리스트" 탭/필터 추가** (이미 active/suspended/deleted 필터·정지/해제 API 존재 → 재활용, 적은 작업)
- 표시: 사유(노쇼누적/채팅위반), banned_at, 해제 버튼(`released` 처리)
- (원하면 별도 섹션으로 분리 가능하나 현재는 회원관리 확장이 일관적)

### 11-6. 재활용 가능한 기존 자산

- `User.is_banned/banned_at/ban_reason`, `is_deleted/deleted_at` + 관리자 정지/해제 API
- `User.contact_warning_count` (채팅 경고 카운트 이미 존재)
- `NoShowReport` + 관리자 노쇼 조회 API
- `Verification`(PASS 본인인증) — `ci_hash`만 추가
