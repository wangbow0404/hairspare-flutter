## 🔴 긴급 — 지금 바로 확인·조치가 필요해요

### 1. 관리자 API 상당수에 인증 체크가 아예 없음

`api-gateway/app/routes/proxy.py`에서 아래 API들은 로그인 여부·관리자 여부를 **전혀 확인하지 않습니다.** 즉 URL만 알면 로그인 없이도 누구나 호출할 수 있어요.

- `POST /api/admin/users/{user_id}/suspend` — 회원 정지
- `POST /api/admin/users/{user_id}/unsuspend` — 정지 해제
- `DELETE /api/admin/users/{user_id}` — 회원 탈퇴 처리
- `DELETE /api/admin/users/{user_id}/permanent` — **회원 영구 삭제** (DB에서 완전히 지움)
- `GET /api/admin/users`, `GET /api/admin/users/{user_id}` — 전체 회원 목록·상세 조회
- `GET /api/admin/stats`, `GET /api/admin/activities`, `GET /api/admin/audit-logs` — 통계·활동로그
- 그 외 `proxy_admin_*`로 auth-service에 그대로 넘기는 다수의 라우트(정산취소, 노쇼신고, 공고, 신고, 결제 등)

넘겨받는 쪽인 `services/auth-service/app/api/routes.py`의 `/api/admin/*` 처리 함수들(35개 확인)도 마찬가지로 관리자 여부를 확인하지 않아요. 게이트웨이에도, 그 뒤 서비스에도 이중으로 체크가 빠진 상태예요.

**쉽게 말하면**: 지금 이 URL 주소들을 알아내면(추측 가능한 패턴이라 어렵지 않음) 로그인 안 한 외부인도 회원을 정지시키거나 영구 삭제할 수 있는 상태예요.

### 2. "관리자 확인" 함수 자체가 인증 실패 시 관리자로 취급함

일부 관리자 API(설정, 지역, 등급, 카테고리 등 8개)는 `_require_admin`이라는 함수로 확인하긴 하는데, 이 함수 내부 로직에 문제가 있어요:

```python
def _admin_chat_resolve_admin_id(request, db):
    # JWT 토큰이 있으면 검사하지만...
    # 토큰이 없거나 잘못됐으면 → DB에서 "가장 먼저 만들어진 관리자 계정"을 그냥 반환해버림
    return _admin_chat_get_admin_id(db)
```

즉 로그인 토큰을 아예 안 보내도, "관리자 계정이 하나라도 있으면" 그 관리자인 것처럼 통과시켜버려요. 사실상 인증 체크가 없는 것과 같은 효과예요.

**권장 조치**: 이 두 가지는 서비스 운영에 직결되는 문제라, 이번 정리와 별도로 빠르게 고치는 걸 추천해요. 원하시면 바로 고쳐드릴게요.

## 🟠 높음 — 운영 환경변수 설정을 꼭 확인해주세요

### 3. 비밀키에 하드코딩된 기본값이 있음

아래 두 값은 환경변수(`JWT_SECRET_KEY`, `INTERNAL_NOTIFY_SECRET`)가 설정 안 돼 있으면 **코드에 그대로 적힌 기본값**을 씁니다(실제 값은 코드에서 직접 확인 가능해서 이 문서엔 옮겨적지 않아요):

- `JWT_SECRET_KEY` — 로그인 토큰(JWT) 서명에 씀. 이 값이 새어나가면 누구나 "나는 이 사람이다"라는 가짜 토큰을 만들 수 있어요.
- `INTERNAL_NOTIFY_SECRET` — 서비스 간 내부 통신 인증에 씀. 특히 아래 두 API가 이 값 하나로만 보호되고 있어요(로그인 체크 자체가 없음, 코드에 명시적으로 예외 처리돼 있음):
  - `DELETE /internal/admin/purge-user` — **이메일로 회원을 영구 삭제**
  - `GET /internal/admin/user-lookup` — 이메일로 회원 개인정보 조회

**이 두 값이 Railway 환경변수에 실제로 강력한 무작위 값으로 설정돼 있는지 꼭 확인해주세요.** 설정 안 돼 있다면, 코드가 저장소에 있는 이상 이 기본값도 사실상 공개된 값이에요. (제가 Railway 환경변수에 직접 접근할 권한은 없어서 이 부분은 직접 확인이 필요해요.)

### 4. 연락처(전화번호) 공유 차단이 앱에서만 검사되고, 서버는 안 봄

지금 채팅 메시지의 "전화번호 같은 걸 보내면 막는" 기능(`ContactBlocker`)은 **Flutter 앱 안에서만** 검사해요. 서버(`POST /api/messages`)는 메시지 내용을 저장할 때 전화번호·이메일 패턴을 전혀 확인하지 않아요.

**쉽게 말하면**: 앱을 그대로 쓰는 일반 사용자는 잘 막히지만, 앱을 변형해서(또는 API를 직접 호출해서) 접근하면 이 차단을 완전히 우회하고 전화번호를 자유롭게 주고받을 수 있어요. 최근에 만든 "3회 적발 시 대화방 삭제" 기능도 클라이언트가 스스로 "저 위반했어요"라고 신고해야만 작동하는 구조라, 우회하면 신고 자체가 안 가요.

## 🟡 중간 — 개선을 권장해요

- **이미지 프록시 SSRF 취약점**: `/api/auth/image-proxy?url=...`가 `.r2.dev/`라는 문자열이 URL 어딘가에 포함되기만 하면 통과시켜요(`https://공격자주소.com/x?y=.r2.dev/` 같은 URL도 통과 가능). 진짜 도메인 검증이 아니라 문자열 포함 여부만 봐서, 서버가 공격자가 지정한 아무 주소나 대신 요청해줄 위험이 있어요.
- **로그인 무차별 대입(brute-force) 방어 없음**: 비밀번호를 계속 틀려도 계정 잠금이나 시도 횟수 제한이 없어요.
- **API 속도 제한 전혀 없음**: 게이트웨이 전체에 요청 횟수를 제한하는 장치가 없어요. 로그인·관리자 API 포함 전부 무제한 호출 가능.
- **네이버 로그인 시크릿 노출**: `ios/Runner/Info.plist`에 네이버 클라이언트 시크릿이 평문으로 커밋돼 있어요(git 이력에 남음). 네이버 SDK 특성상 완전히 피하긴 어렵지만, 유출 우려가 있다면 네이버 개발자센터에서 시크릿을 재발급받는 걸 고려해보세요.

---

## 🟢 낮음 — 여유 있을 때 손보면 좋아요

- **에러 메시지 노출**: `proxy.py`의 관리자 API 39곳이 실패 시 `f"실패: {e}"` 형태로 파이썬 예외 메시지를 그대로 클라이언트에 돌려줘요. DB 컬럼명 등 내부 정보가 섞여 나올 수 있어요.
- **파일 업로드 타입 검증 없음**: 이미지·동영상 업로드는 용량 제한(10MB/100MB)만 있고, 실제 파일 형식은 검증하지 않아요. 클라이언트가 "이건 이미지예요"라고 우기면 그대로 믿어요.
- **HTTPS 강제가 코드에 없음**: Railway가 알아서 처리해주는 부분에 의존하고 있어요.
- **인증서 고정(certificate pinning)·탈옥/루팅 탐지 없음**: 요즘 앱들이 종종 쓰는 추가 보안 장치인데, 지금은 없어요. 필수는 아니고, 나중에 여유 되면 고려할 부분.

## ✅ 잘 되어 있는 부분

- **SQL 인젝션**: 샘플로 확인한 50곳 이상의 raw SQL이 전부 파라미터 바인딩(`:변수명` 방식)을 정확히 쓰고 있어요. 문자열을 직접 이어붙여 쿼리를 만드는 위험한 코드는 못 찾았어요.
- **비밀번호**: bcrypt로 해싱해서 저장하고, 평문으로 로그에 남기지도 않아요.
- **토큰 저장(앱)**: 로그인 토큰은 iOS Keychain·Android Keystore를 쓰는 `flutter_secure_storage`에 저장돼요(더 안전한 방식). 리프레시 토큰은 쿠키 방식이라 앱 폴더 안 파일로 저장되는데, 이건 이 방식(HttpOnly 쿠키)을 쓰는 앱들의 일반적인 구조예요.
- **`.env` 관리**: 진짜 비밀값이 든 `.env` 파일은 git에 커밋된 적이 없어요. 저장소엔 값이 비어있는 예시 파일(`.env.example`)만 있어요.
- **CORS**: 전체 허용(`*`)이 아니라 `hairspare.co.kr` 등 정해진 도메인만 허용해요.
- **카카오 로그인 키**: 카카오 네이티브 앱 키는 코드에 있지만, 이건 카카오 SDK 설계상 원래 공개돼도 되는 키(전화번호부의 "번호"같은 역할)라 문제 없어요.
