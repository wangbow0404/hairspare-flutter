# HairSpare 서비스 개발 전체 과정 대시보드 (재정립)

> 작성 목적: 사용자 제공 프로세스 이미지(기획→설계→개발→소스관리→배포→출시→유지보수(+수익화))를 기준으로,
> 현재 HairSpare 코드베이스/문서 상태를 다시 맞춰 "무엇이 끝났고", "무엇이 진행 중이며", "다음에 무엇을 할지"를 실행 가능하게 정리한다.
>
> 기준 근거: `DEVELOPER_BRIEFING.md`, `ARCHITECTURE_REFACTOR.md`, 현재 코드 상태(최근 반영: 챌린지 immersive, 근무체크 종료시간 기준, 전역 라우터/appRouter, mock 비디오 재생)

---

## 0) 한눈에 보는 현재 위치

- 현재 단계 포지션: **개발 후반 + 소스관리/배포 체계 정비 초입**
- 강점: 앱 아키텍처(UI→ViewModel→Service), 라우팅(go_router), DI(get_it), 도메인 화면 구현 폭
- 약점: 소스관리 정책(브랜치/릴리즈), 배포 파이프라인, 출시 체크리스트, 운영/관측, 수익화 KPI 체계
- 핵심 결론: 이제 "기능 추가"와 병행해서 **운영 가능한 제품 체계**를 구축해야 함

## 1.4 소스관리 (이미지의 "소스관리")

### 현재 상태
- 상태: **낮음~중간 (45%)**
- 현재 관찰
  - 기능 개발은 활발하지만 브랜치/릴리즈 규칙의 문서화가 약함
  - 커밋/PR 템플릿, 라벨 전략, 태깅 전략이 명확하지 않음

### 반드시 정립할 것
1. 브랜치 전략
   - `main`: 출시 가능 상태만
   - `develop`(선택) 또는 short-lived feature branch
   - 브랜치 네이밍: `feat/`, `fix/`, `refactor/`, `docs/`, `chore/`
2. PR 규칙
   - PR 본문 템플릿(요약/변경점/테스트/리스크)
   - 리뷰 최소 1인 + 체크리스트 통과
3. 릴리즈 태깅
   - `vX.Y.Z` 태그 기준
   - 배포 대상 커밋 명시
4. 핫픽스 흐름
   - `hotfix/*` 브랜치 + 릴리즈 노트 반영

### 산출물
- `SOURCE_CONTROL_POLICY.md`
- `.github/pull_request_template.md`
- `RELEASE_TAGGING_POLICY.md`

## 1.5 배포 (이미지의 "배포")

### 현재 상태
- 상태: **초기 (35%)**
- 현재 관찰
  - 로컬/개발 중심 운영
  - 환경 분리(dev/stage/prod)와 자동화 파이프라인이 약함

### 정립할 것
1. 환경 분리
   - `dev`, `staging`, `prod` 설정 분리
   - 비밀값은 dart-define/CI secret 관리
2. CI 파이프라인
   - `flutter pub get` → `dart analyze` → `flutter test`
   - 빌드 산출물(apk/ipa) 생성 단계
3. CD 전략
   - 내부 테스트 배포(TestFlight/Internal track)
   - 승인 후 프로덕션 배포
4. 배포 체크리스트
   - 버전/빌드번호, 환경변수, 앱 권한, 크래시 SDK

### 산출물
- `DEPLOYMENT_PIPELINE_V1.md`
